## 2. Build Pipeline

### 2a. `build.rs` — WASM → C++ Transpilation

**File**: `moPro/mopro-example-app/build.rs`

Runs automatically before `cargo build`. Two phases:

**Phase 1 — Transpile circuit WASM to C++**:
```rust
rust_witness::transpile::transpile_wasm("./test-vectors/circom".to_string());
```
Walks every `*.wasm` in `test-vectors/circom/` and converts each WASM witness calculator (compiled from the original `.circom` source offline) into a `.cpp` file using the w2c2 WASM-to-C transpiler. Example: `sha256_16.wasm` → `sha256_16.cpp`.

**Phase 2 — Compile C++ into static library**:
```rust
let mut build = cc::Build::new();
build.cpp(true);
build.define("W2C2_LOOP_START", Some(""));
// collects all .cpp files from test-vectors/circom/witnesscalc/
build.compile("witnesses");
println!("cargo:rustc-link-lib=static=witnesses");
```
Produces `libwitnesses.a` which gets statically linked into `libmopro_example_app.so`.

> **Key insight**: The `.wasm` files are **build-time inputs only**. They are not shipped to the device. At runtime, witness calculation is pure native C++.

### 2b. `mopro_ffi::app!()` — What the Macro Generates

**File**: `moPro/mopro-example-app/src/lib.rs:8`

This proc macro does three things at compile time:

**1. Generates a C ABI wrapper for every `#[uniffi::export]` function.**

For `generate_circom_proof`, it emits (conceptually):
```c
// Symbol name format: ffi_{crate_name}_fn_func_{function_name}
// No Rust name mangling. This is what JNA dlsym-calls.
RustBuffer ffi_mopro_example_app_fn_func_generate_circom_proof(
    RustBuffer zkey_path,
    RustBuffer circuit_inputs,
    int8_t proof_lib,
    UniffiRustCallStatus* out_status
);
```

**2. Generates Rust-side dispatcher stubs** — each stub deserializes `RustBuffer` args via `FfiConverter::try_lift()`, calls your actual Rust function, serializes the return value back via `FfiConverter::lower()`, and writes to `UniffiRustCallStatus`.

**3. Embeds `uniffi-bindgen` metadata** into the compiled binary. The `uniffi-bindgen` tool reads this metadata section to generate `mopro.kt` and `mopro.swift` — it needs only the compiled `.so`/`.a`, not the Rust source.

### 2c. Circuit Registration Macros

Two compile-time registries are built in `lib.rs`:

**Groth16 witness registry** (`src/lib.rs`, requires `groth16` feature):
```rust
rust_witness::witness!(sha25616);     // generates sha25616_witness fn ptr → compiled C++
rust_witness::witness!(keccak25616);
// ... 28 circuits total

set_groth16_circuits! {
    ("sha256_16.zkey", WitnessFn::RustWitness(sha25616_witness)),
    ("keccak256_16.zkey", WitnessFn::RustWitness(keccak25616_witness)),
    // ...
}
// Expands to: static GROTH16_CIRCUITS: &[(&str, WitnessFn)] = &[...];
// And: pub fn groth16_get(name: &str) -> Option<WitnessFn>
```

### 2d. `mopro build` and `mopro update`

`mopro build` (controlled by `Config.toml`) runs `cargo build --release --target aarch64-linux-android` etc., then runs `uniffi-bindgen generate` to produce `mopro.kt` and `mopro.swift`.

`mopro update` copies:
- **Android**: `libmopro_example_app.so` → `flutter/mopro_flutter_plugin/android/src/main/jniLibs/{arch}/`
- **Android**: `mopro.kt` → `flutter/mopro_flutter_plugin/android/src/main/kotlin/uniffi/mopro/mopro.kt`
- **iOS**: `MoproBindings.xcframework` → `flutter/mopro_flutter_plugin/ios/Frameworks/`
- **iOS**: `mopro.swift` (renamed to `MoproCircom.swift` or `MoproNoir.swift`) → `flutter/mopro_flutter_plugin/ios/Classes/`

**iOS config-swap**: Because Swift has global symbol scope, each proof system gets a separate two-pass build. `src/configs/lib_groth16.rs` is copied to `src/lib.rs`, built, then `src/configs/lib_barretenberg.rs` is copied and built again. Each pass produces a different `mopro.swift` with prefixed type names (`Groth16RustBuffer`, `BarretenbergRustBuffer`) to prevent symbol collisions.

## 3. The FFI Binding Layer

### 3.1 `mopro_ffi::app!()` — What the Macro Generates

See [§2b above](#2b-mopro_ffiapp--what-the-macro-generates).

### 3.2 `RustBuffer` — The Universal Transport

**Everything** that crosses the FFI boundary — strings, structs, lists, error messages — is packed into a `RustBuffer`.

```
RustBuffer layout (24 bytes on arm64):
  [0..8]  capacity: u64   — total allocated bytes
  [8..16] len: u64        — bytes actually written
  [16..24] data: *mut u8  — pointer to Rust heap
```

In Kotlin (`mopro.kt:42`):
```kotlin
@Structure.FieldOrder("capacity", "len", "data")
open class RustBuffer : Structure() {
    @JvmField var capacity: Long = 0
    @JvmField var len: Long = 0
    @JvmField var data: Pointer? = null

    companion object {
        // Allocate: must go through Rust's allocator
        internal fun alloc(size: ULong) = uniffiRustCall() { status ->
            UniffiLib.INSTANCE.ffi_mopro_example_app_rustbuffer_alloc(size.toLong(), status)
        }
        // Free: must go through Rust's allocator
        internal fun free(buf: RustBuffer.ByValue) = uniffiRustCall() { status ->
            UniffiLib.INSTANCE.ffi_mopro_example_app_rustbuffer_free(buf, status)
        }
    }
}
```

**Critical rule**: the heap pointed to by `data` is **always allocated by Rust's allocator** and **must be freed by calling back into the `.so`**. The JVM cannot `free()` this memory directly — doing so would cause a heap corruption crash because JVM's allocator and Rust's allocator are separate.

In Swift (`MoproCircom.swift:14`), the same struct exists as `CircomRustBuffer` (prefixed to prevent Swift symbol collisions across the two proof system builds):
```swift
fileprivate extension CircomRustBuffer {
    static func from(_ ptr: UnsafeBufferPointer<UInt8>) -> CircomRustBuffer {
        try! rustCall { ffi_deimos_circom_rustbuffer_from_bytes(CircomForeignBytes(bufferPointer: ptr), $0) }
    }
    func deallocate() {
        try! rustCall { ffi_deimos_circom_rustbuffer_free(self, $0) }
    }
}
```

### 3.3 `FfiConverter` — Serialization Protocol

Every type that crosses the FFI boundary has a generated `FfiConverter`. The interface (`mopro.kt:142`):

```kotlin
interface FfiConverter<KotlinType, FfiType> {
    fun lower(value: KotlinType): FfiType      // Kotlin → wire format (outbound call)
    fun lift(value: FfiType): KotlinType       // wire format → Kotlin (return value)
    fun write(value: KotlinType, buf: ByteBuffer)   // serialize into stream
    fun read(buf: ByteBuffer): KotlinType            // deserialize from stream
    fun allocationSize(value: KotlinType): ULong     // bytes to pre-allocate
}
```

The wire format is **big-endian binary — not JSON**:

| Kotlin/Swift type | Wire encoding |
|---|---|
| `String` | `[u32 byte_len][utf-8 bytes]` |
| `List<T>` | `[u32 count][T T T ...]` |
| `Map<K,V>` | `[u32 count][[K][V] [K][V] ...]` |
| `Boolean` | `u8` (0 or 1) |
| `UInt` (u32) | `u32 big-endian` |
| `ByteArray` / `Vec<u8>` | `[u32 byte_len][bytes]` |
| Enum variant | `u8` discriminant |
| Struct / Record | fields written in declaration order |
| `Option<T>` | `u8` (0=None, 1=Some) + `[T]` if Some |

For `CircomProofResult`, the full binary layout looks like:
```
[CircomProof]
  [G1 a]
    [u32 x_len][x_utf8]    -- BN254 field element as decimal string e.g. "2188824..."
    [u32 y_len][y_utf8]
    [u32 z_len][z_utf8]
  [G2 b]
    [u32 x_count=2][x[0]][x[1]]   -- G2 has two field elements per coordinate
    [u32 y_count=2][y[0]][y[1]]
    [u32 z_count=2][z[0]][z[1]]
  [G1 c]
    ...
  [u32 protocol_len]["groth16"]
  [u32 curve_len]["BN254"]
[u32 inputs_count]
  [u32 input0_len][input0_utf8]
  ...
```

In Swift, the reader is a `(data: Data, offset: Data.Index)` tuple threaded through every `read()` call — a cursor advancing through the binary stream (`MoproCircom.swift:75`):
```swift
fileprivate func readInt<T: FixedWidthInteger>(
    _ reader: inout (data: Data, offset: Data.Index)
) throws -> T {
    let range = reader.offset..<reader.offset + MemoryLayout<T>.size
    var value: T = 0
    let _ = withUnsafeMutableBytes(of: &value) { reader.data.copyBytes(to: $0, from: range) }
    reader.offset = range.upperBound
    return value.bigEndian
}
```

### 3.4 `UniffiRustCallStatus` — Error Channel

```kotlin
// mopro.kt:219
@Structure.FieldOrder("code", "error_buf")
internal open class UniffiRustCallStatus : Structure() {
    @JvmField var code: Byte = 0                           // 0=success, 1=Err, 2=panic
    @JvmField var error_buf: RustBuffer.ByValue = RustBuffer.ByValue()
}
```

Every C ABI call receives a **pointer to this** as its last argument. Rust writes the outcome before returning. The caller pattern (`mopro.kt:264`):

```kotlin
private inline fun <U, E: Exception> uniffiRustCallWithError(
    errorHandler: UniffiRustCallStatusErrorHandler<E>,
    callback: (UniffiRustCallStatus) -> U
): U {
    var status = UniffiRustCallStatus()
    val return_value = callback(status)          // ← JNA call happens here
    uniffiCheckCallStatus(errorHandler, status)  // ← inspect after return
    return return_value
}

// Inspection:
if (status.isSuccess()) return                  // code == 0, happy path
else if (status.isError()) {
    throw errorHandler.lift(status.error_buf)   // deserialize MoproError
} else if (status.isPanic()) {
    throw InternalException(FfiConverterString.lift(status.error_buf))
}
```

Rust's `Result<T, MoproError>`:
- `Ok(value)` → sets `code = 0`, serializes `value` into return `RustBuffer`
- `Err(e)` → sets `code = 1`, serializes `e` (via `FfiConverterTypeMoproError::lower`) into `error_buf`
- `panic!` → sets `code = 2`, serializes panic message string into `error_buf`

This is how the Rust `Result` type becomes exceptions in Kotlin/Swift.

### 3.5 JNA — How Kotlin Calls the `.so` (Android)

JNA (Java Native Access) bridges JVM bytecode to native C functions without writing JNI glue code. How it loads (`mopro.kt:370`):

```kotlin
private fun findLibraryName(componentName: String): String = "mopro_example_app"
// → Native.load("mopro_example_app", UniffiLib::class.java)
// → System.loadLibrary("mopro_example_app")
// → loads jniLibs/arm64-v8a/libmopro_example_app.so
```

The `UniffiLib` interface maps Kotlin method signatures to C symbols by name:
```kotlin
internal interface UniffiLib : Library {
    // Memory management (always present)
    fun ffi_mopro_example_app_rustbuffer_alloc(
        size: Long, uniffi_out_err: UniffiRustCallStatus.ByReference
    ): RustBuffer.ByValue

    fun ffi_mopro_example_app_rustbuffer_free(
        buf: RustBuffer.ByValue, uniffi_out_err: UniffiRustCallStatus.ByReference
    )

    // Proving function
    fun ffi_mopro_example_app_fn_func_generate_circom_proof(
        zkey_path: RustBuffer.ByValue,
        circuit_inputs: RustBuffer.ByValue,
        proof_lib: Byte,
        uniffi_out_err: UniffiRustCallStatus.ByReference
    ): RustBuffer.ByValue

    companion object {
        internal val INSTANCE: UniffiLib by lazy {
            loadIndirect<UniffiLib>(componentName = "mopro-example-app")
        }
    }
}
```

**How JNA marshals `Structure` subclasses**:

`@Structure.FieldOrder("capacity", "len", "data")` defines the C struct layout. JNA:
1. Allocates native (off-heap) memory matching the struct's field sizes
2. Copies Java field values byte-for-byte into native memory
3. Passes the native pointer to the C function
4. After the call, reads native memory back into Java field values

`Structure.ByValue` — struct passed by value (copied onto the stack). Used for `RustBuffer` (24 bytes, cheap to copy).
`Structure.ByReference` — pointer to struct passed. Used for `UniffiRustCallStatus` (Rust must write to it).

**No JNI `native` methods needed** — JNA handles all the `dlsym` and calling convention machinery at runtime via reflection.

### 3.6 C Headers — How Swift Calls the `.a` (iOS)

On iOS, the XCFramework bundle (`MoproBindings.xcframework`) contains:
```
ios-arm64/
  Headers/
    deimos_circom.h        -- C function declarations
    deimos_circom_shared.h -- shared type declarations
  libmopro_example_app.a  -- static library
```

Swift imports these directly — Swift can call C functions declared in C headers without any bridging layer. The generated `MoproCircom.swift` calls:
```swift
// ffi_deimos_circom_fn_func_generate_circom_proof is declared in deimos_circom.h
let result = try rustCall { status in
    ffi_deimos_circom_fn_func_generate_circom_proof(
        FfiConverterString.lower(zkeyPath),
        FfiConverterString.lower(circuitInputs),
        FfiConverterTypeCircomProofLib.lower(proofLib),
        status
    )
}
```

**Threading difference**:
- **iOS** (`MoproFlutterPlugin.swift:142`): Uses `DispatchQueue.global(qos: .userInitiated).async` — FFI runs on a background thread, result dispatched back to main.
- **Android** (`MoproFlutterPlugin.kt:139`): No threading — `generateCircomProof(...)` is called synchronously on the platform (UI) thread. Long proofs will block the main thread.

## 5. Proof System Flows

### 5.1 Groth16 (Arkworks/Circom)

**Feature flag**: `groth16` (enabled by default)

**Circuit registration** (compile-time):
```rust
// src/lib.rs (gated with #[cfg(feature = "groth16")])
rust_witness::witness!(sha25616);   // → fn ptr to compiled C++ from sha256_16.wasm

set_groth16_circuits! {
    ("sha256_16.zkey", WitnessFn::RustWitness(sha25616_witness)),
}
// expands to: static GROTH16_CIRCUITS + fn groth16_get(name) -> Option<WitnessFn>
```

**Runtime flow**:
```
User: Framework=Groth16, Circuit=sha256_16, Input=16 bytes
  → input JSON: {"in": ["0","1",...,"15"]}
  → zkey asset extracted to filesystem: getApplicationDocumentsDirectory()/sha256_16.zkey
  → generateGroth16Proof(zkeyPath, inputsJson, ProofLib.ARKWORKS)
      → groth16_get("sha256_16.zkey") → sha25616_witness
      → sha25616_witness(inputs) → Vec<FieldElement> [native C++]
      → ProvingKey loaded from .zkey
      → Groth16::prove(pk, witness) → Proof{a:G1, b:G2, c:G1}
      → Groth16ProofResult{proof, inputs: Vec<String>}
```

**Supported circuits** (28 total):

| Hash | Input sizes |
|---|---|
| blake2s256 | 16, 32, 64, 128 bytes |
| blake3 | 16, 32, 64, 128 bytes |
| keccak256 | 16, 32, 64, 128 bytes |
| sha256 | 16, 32, 64, 128 bytes |
| pedersen | 16, 32, 64, 128 bytes |
| MiMC256 | 16, 32, 64, 128 field elements |
| Poseidon | 16, 32, 64, 128 field elements |
| Rescue Prime | 16, 32, 64, 128 field elements |

**Proof backends**:
- `ProofLib.ARKWORKS` — pure Rust Groth16 (works everywhere)
- `ProofLib.RAPIDSNARK` — C++ prover (faster on Android, native `.so` dependency)

### 5.2 Barretenberg (Noir / Ultra Honk)

**Feature flag**: `barretenberg` (enabled by default)

> **Historical note**: The Halo2 backend (Plonk/HyperPlonk/Gemini Fibonacci circuits) was removed as unused dead code. Groth16 and Barretenberg are the primary proving systems.

**Source**: `src/barretenberg.rs` — wraps `noir_rs` crate.

### Barretenberg: Noir / Ultra Honk

**Circuit format**: `noir_multiplier2.json` contains ACIR bytecode:
```json
{ "bytecode": "H4sI..." }   // base64-gzipped ACIR
```

**Lifecycle**:
```
First call (expensive):
  getBarretenbergVerificationKey(circuitPath, srsPath, onChain=true, lowMemoryMode)
    → get_bytecode(circuitPath): read JSON, extract "bytecode" field, decode
    → setup_srs_from_bytecode(bytecode, srsPath)
    → derive verification key
    → Vec<u8>   (store this — generation is slow)

Prove:
  generateBarretenbergProof(circuitPath, srsPath, inputs, onChain=true, vk, lowMemoryMode)
    → on_chain=true  → prove_ultra_honk_keccak()   (Solidity-verifiable)
    → on_chain=false → prove_ultra_honk_poseidon()  (faster, off-chain)
    → Vec<u8> proof bytes

Verify:
  verifyNoirProof(circuitPath, proof, onChain, vk, lowMemoryMode)
    → verify_ultra_honk_keccak/poseidon(proof, vk)
    → bool
```

`on_chain=true` uses **Keccak** in the proof transcript — Solidity verifiers use Keccak for hashing, so this makes the proof directly verifiable on-chain.
`on_chain=false` uses **Poseidon** — algebraically friendly, roughly 2× faster but only verifiable in ZK-native contexts.

**FFI type**: proof is `Vec<u8>` (raw bytes), VK is `Vec<u8>`. No struct wrapping.

### 5.4 RISC0 zkVM

Structurally different from all other proof systems — it proves **arbitrary program execution**, not a fixed circuit.

**Guest program** (`risc0-circuit/src/main.rs`):
```rust
fn main() {
    let input: u32 = env::read();
    let output = input * input;   // (or any computation)
    env::commit(&output);
}
```

**Build**: `risc0-build` (in `risc0-circuit/build.rs`) compiles the guest to RISC-V ELF and embeds it at compile time:
```rust
// risc0-circuit/build.rs
risc0_build::embed_methods();
// → generates: methods/src/lib.rs with:
//   pub const RISC0_CIRCUIT_ELF: &[u8] = include_bytes!("...");
//   pub const RISC0_CIRCUIT_ID: [u32; 8] = [...];
```

**Runtime**:
```rust
pub fn risc0_prove(input: u32) -> Result<Risc0ProofOutput, Risc0Error> {
    let env = ExecutorEnv::builder().write(&input)?.build()?;
    let prover = default_prover();   // RISC0 STARK prover
    let receipt = prover.prove(env, RISC0_CIRCUIT_ELF)?;
    let receipt_bytes = bincode::serialize(&receipt)?;
    Ok(Risc0ProofOutput { receipt: receipt_bytes })
}

pub fn risc0_verify(receipt_bytes: Vec<u8>) -> Result<Risc0VerifyOutput, Risc0Error> {
    let receipt: Receipt = bincode::deserialize(&receipt_bytes)?;
    receipt.verify(RISC0_CIRCUIT_ID)?;
    let output_value: u32 = receipt.journal.decode()?;
    Ok(Risc0VerifyOutput { is_valid: true, output_value })
}
```

**FFI types**: `input: u32` (passed as `Int` in Kotlin/Swift), `receipt: Vec<u8>` (bincode-serialized STARK receipt, several KB).

### 5.7 IMP1 (External AAR — Bypasses UniFFI)

IMP1 from Ingonyama is a hardware-accelerated Groth16 prover. It does **not** go through the UniFFI/mopro stack. It uses a separate Flutter MethodChannel:

**Channel**: `com.deimos.imp1/prover`
**File**: `flutter/android/app/src/main/kotlin/.../IMP1ProverChannel.kt`

```kotlin
// Different channel, completely separate from mopro_flutter
MethodChannel("com.deimos.imp1/prover").setMethodCallHandler { call, result ->
    when (call.method) {
        "generateProof" -> {
            val circuitName = call.argument<String>("circuitName")
            // Copy .wtns (pre-computed witness) and .zkey from flutter_assets/
            NativeBridge.prove(witnessPath, zkeyPath, proofPath, publicPath, CPU)
            result.success(mapOf("proof" to proofJson, "provingTimeMs" to ...))
        }
    }
}
```

**Key difference from Circom flow**: IMP1 requires **pre-computed witness files** (`.wtns`). The witness is not computed at runtime — it must be bundled as an asset. This means IMP1 can't vary inputs at runtime (or must pre-compute witnesses offline for each input set).

**Dependencies**: `app/libs/imp1-0.2.2.aar` (local AAR), `libc++_shared.so` (packed in APK via `packagingOptions`).

## 6. Flutter Plugin Layer

### 6.1 MethodChannel Protocol

All communication (except IMP1) goes through a single Flutter `MethodChannel("mopro_flutter")`. The codec is `StandardMethodCodec` — Flutter's binary codec that handles: `null`, `bool`, `int`, `double`, `String`, `Uint8List`, `List`, `Map`.

**Dart side** (`mopro_flutter_method_channel.dart`):
```dart
final methodChannel = const MethodChannel('mopro_flutter');
await methodChannel.invokeMethod<Map>('generateGroth16Proof', {
    'zkeyPath': path,
    'inputs': jsonString,
    'proofLib': 0,    // int, not enum
});
```

**Platform side**: Both `MoproFlutterPlugin.kt` and `MoproFlutterPlugin.swift` register with the same channel name.

### 6.2 Asset Extraction to Filesystem

**Why necessary**: Native code needs filesystem paths. Flutter assets live in the APK/IPA bundle — read-only and not directly path-addressable by native code on Android. They must be extracted.

**Dart** (`mopro_flutter.dart`):
```dart
Future<String> copyAssetToFileSystem(String assetPath) async {
    final byteData = await rootBundle.load(assetPath);    // load from bundle
    final directory = await getApplicationDocumentsDirectory();
    final file = File('${directory.path}/${basename(assetPath)}');
    await file.writeAsBytes(byteData.buffer.asUint8List());
    return file.path;   // now a real filesystem path
}
```

This is called before every `generateCircomProof`, `generateHalo2Proof`, etc. For large `.zkey` files (100MB+), this is the bottleneck — the file is re-extracted every call unless cached.

**iOS asset resolution** (`MoproFlutterPlugin.swift:127`):
```swift
let key = FlutterDartProject.lookupKey(forAsset: zkeyPath)
// key = "flutter_assets/test-vectors/circom/sha256_16.zkey"
if let bundlePath = Bundle.main.path(forResource: key, ofType: nil) {
    resolvedZkeyPath = bundlePath  // direct bundle path — no copy needed on iOS
}
```

iOS can access bundle files directly by path — no extraction needed. The `lookupKey` converts Dart asset paths to iOS bundle resource paths.

### 6.3 Type Definitions (`mopro_types.dart`)

```dart
class G1Point { final String x, y, z; }
class G2Point { final List<String> x, y, z; }

    final List<String> inputs;

    Map<String, dynamic> toMap() { ... }   // for verifyCircomProof
}

class Halo2ProofResult { final Uint8List proof, inputs; }
```

## 8. Memory Ownership Rules

| Object | Allocated by | Freed by | Notes |
|--------|-------------|---------|-------|
| `RustBuffer` args (outbound) | Kotlin/Swift via `RustBuffer.alloc()` → calls into `.so` | Rust, when `try_lift()` takes ownership | Must use `ffi_..._rustbuffer_alloc`, not `malloc` |
| `RustBuffer` return value | Rust | Kotlin/Swift via `liftFromRustBuffer()` → calls `RustBuffer.free()` → into `.so` | Auto-freed in `liftFromRustBuffer` finally block |
| `error_buf` in `UniffiRustCallStatus` | Rust | Kotlin/Swift via `errorHandler.lift(error_buf)` | lift() calls free() as part of deserialization |
| `.zkey` file data | Rust (`std::fs::read` internal buffer) | Rust (dropped at end of `prove()`) | Never in JVM heap |
| `Vec<u8>` proof bytes | Rust | Serialized into `RustBuffer` → freed with buffer | |
| `libwitnesses.a` symbols | Static link (process lifetime) | Process exit | C++ witness functions |
| RISC0 receipt | Rust (bincode-serialized) | Serialized into `RustBuffer` → freed with buffer | Receipt can be 100KB+ |

**Golden rule**: Memory allocated by Rust's allocator must be freed by calling back into the `.so`. Kotlin's JVM GC and Rust's allocator are completely separate. Cross-allocator free = crash.

## 9. Backend Naming Convention

All backends in this workspace are named after their **proving system**, not the circuit language used to write the circuits:

| Canonical name | Proving system | Circuit language | Rust source |
|----------------|---------------|-----------------|-------------|
| `groth16` | Groth16 (Arkworks) | Circom | `src/groth16.rs` |
| `barretenberg` | UltraHonk (Barretenberg) | Noir | `src/barretenberg.rs` |
| `cairo_m` | STARKs / Stwo | Cairo-M | `cairo-m-prover/src/lib.rs` |
| `provekit` | ProveKit | Noir (accelerated) | `provekit-wrapper/src/lib.rs` |
| `risc0` | RISC0 zkVM | RISC-V assembly | `risc0-circuit/src/main.rs` |

**Rationale**: a circuit language can target multiple proving systems (e.g. Noir compiles to both Barretenberg UltraHonk and ProveKit). Naming by proving system makes the backend identity unambiguous.

## 10. Feature Flags

Feature flags in `moPro/mopro-example-app/Cargo.toml` control which backends are compiled in:

| Feature flag | Crates enabled | Proving system | Default |
|---|---|---|---|
| `groth16` | `circom-prover`, `rust-witness`, `num-bigint` | Groth16 via Arkworks | ✅ yes |
| `barretenberg` | `noir_rs` | UltraHonk via Barretenberg | ✅ yes |
| `cairo_m` | `cairo-m-prover` | STARKs via Stwo over M31 | ✅ yes |
| `provekit` | `provekit-wrapper` | ProveKit accelerated Noir | ✅ yes |
| _(always on)_ | `risc0-zkvm`, `methods` | RISC0 zkVM (STARK) | always |

To build with only the Groth16 and RISC0 backends (e.g. for a smaller binary):
```bash
cargo build --no-default-features --features groth16
```

To build all backends (default):
```bash
cargo build
```
